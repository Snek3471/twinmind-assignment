# TwinMind — Live AI Meeting Copilot

Real-time meeting assistant. Three-panel dark UI: live transcript stream, rolling suggestion batches, and a streamed chat panel grounded in the conversation.

## Setup

```bash
npm install
npm run dev
# Open http://localhost:3000
# Click "Add API Key" → paste your Groq key
```

Get a free Groq API key at https://console.groq.com.

## Stack

| Layer | Choice | Why |
|-------|--------|-----|
| Framework | Next.js App Router | API routes run server-side — Groq key never touches the client |
| Styling | Tailwind CSS | Utility-first; no stylesheet context-switching during rapid iteration |
| Transcription | Groq Whisper Large V3 | Sub-2s latency on 30s audio chunks; handles overlapping speech well |
| Suggestions + Chat | Groq `openai/gpt-oss-120b` | Low latency, reliable JSON output, native streaming support |
| Schema validation | Zod | Validates API inputs and LLM JSON output at the same boundary |
| Audio capture | Web MediaRecorder API | Native browser, zero dependencies |

## Prompt Strategy

Suggestions use a **type-based system** — each suggestion must declare itself as one of `QUESTION`, `ANSWER`, `FACT_CHECK`, `TALKING_POINT`, or `CLARIFY`. This forces the model to identify *what kind of help is needed*, not just which topic is relevant. A generic suggestion about "pricing strategy" is less actionable than one typed `TALKING_POINT: Frame the ROI before quoting the number`. The type also determines how the expanded answer is framed in chat.

Each `preview` must deliver **standalone value** — it cannot be a teaser that requires clicking through. If the preview isn't useful on its own, it's not doing its job. The `detail_prompt` field is generated by the model alongside the suggestion: the suggestion itself writes the question it should answer, so the expanded response is always targeted rather than generic.

A `{PREVIOUS_SUGGESTIONS}` placeholder is injected into the prompt before each call, filled with all previews from prior batches. This prevents the model from repeating the same suggestions across refreshes, which is the most common failure mode in rolling-window prompting.

Suggestions use a shorter context window than chat (configurable, default 400 vs 2000 words) because suggestion quality degrades with too much context — the model picks generalities instead of the specific thing happening *right now*. Chat uses the full recent window because grounding a direct question requires more history.

## Audio Chunking

`useTranscript` restarts `MediaRecorder` every 30 seconds rather than using `timeslice`. Each restart produces a complete, self-contained WebM file with a valid header — Whisper can decode it independently without needing prior segments. Timesliced chunks share a header with the first segment; any chunk sent in isolation produces a decode error.

`flushCurrent()` handles the manual refresh button: it cancels the pending auto-restart timer, stops the recorder mid-chunk (which fires `ondataavailable` with all buffered audio), awaits transcription, then restarts the recorder. This means the user gets fresh context immediately on demand — the next suggestion batch runs against transcript that includes everything said up to the moment they clicked.

## Tradeoffs

- **Session-only state**: no DB, no auth. Refresh loses everything. Export button saves JSON.
- **API key in localStorage**: suitable for personal/local use; not for multi-tenant deployment.
- **LLM JSON parsing**: three-stage fallback (array regex → direct parse → bracket-counting object extraction) handles markdown-fenced and partially malformed responses. Zod validates each item individually so one bad object doesn't drop the whole batch.
- **30s chunk latency**: users wait up to 30s for the first chunk. Shorter intervals increase Whisper call volume; silence detection would make this manageable but adds complexity.

## What I'd Improve With More Time

- **Speaker diarization** so suggestions know who said what — `QUESTION` from the other party vs. your own statement require different responses
- **Streaming suggestion output** instead of buffering full JSON — first card visible in ~500ms instead of ~2s
- **Voice activity detection** to skip silent chunks entirely, eliminating the cost and latency of transcribing dead air
- **Shorter chunk interval (10s)** paired with silence detection — finer-grained transcript updates mean suggestions track the conversation more closely
